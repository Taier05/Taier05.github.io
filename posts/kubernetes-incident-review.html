一次 Kubernetes Pod Pending 的完整排查

从事件、调度器到节点资源，沿着证据链定位一个容易被忽略的配置错误。

现象与背景

某次发布后，一个 Deployment 的新 Pod 长时间停留在 Pending。集群中的其他工作负载正常，节点也没有明显告警。只看 Pod 状态，很容易把问题归因于“资源不足”。

样本说明

这是一篇用于展示博客排版的示例文章，命令输出与资源名称均为演示数据。

建立证据链

第一步不是猜，而是看调度事件。事件会告诉我们控制面已经做过什么，以及失败发生在哪个阶段。

kubectl -n demo describe pod log-generator-7c8f7d9d6f-x2k8m

Events:
  Type     Reason            Age   From               Message
  ----     ------            ----  ----               -------
  Warning  FailedScheduling  42s   default-scheduler  no nodes available

事件显示 Pod 已经进入调度阶段，但这还不够。继续对照 Deployment 模板和集群中已注册的调度器，发现模板里存在一个不寻常字段。

检查项现场结果判断
节点 Ready 状态全部 Ready节点整体可用
可分配资源CPU、内存均有余量不是资源耗尽
schedulerNamedaefault-scheduler名称拼写异常
已部署调度器default-scheduler找不到指定调度器

根因判断

Pod 模板把 schedulerName 错写为 daefault-scheduler。Kubernetes 会等待与这个名称匹配的调度器处理 Pod；集群里并不存在该调度器，因此 Pod 一直无法被绑定到节点。

这也解释了为什么扩容节点没有效果：资源供给不是瓶颈，真正中断的是“谁来调度”这一步。

修复与验证

正式修改前，先通过服务端 dry-run 检查字段路径和 API 是否接受补丁：

kubectl -n demo patch deployment log-generator \
  --type='json' \
  -p='[{"op":"replace","path":"/spec/template/spec/schedulerName","value":"default-scheduler"}]' \
  --dry-run=server

dry-run 通过后再执行真实修改，并观察 Deployment 滚动发布、Pod 调度事件和最终就绪状态。验证不能只停留在“命令返回成功”，还应确认新 Pod 已分配节点、容器 Ready，并且业务探针恢复正常。

经验总结

  1. 先读事件，再决定向资源、网络、存储还是控制面继续深入。
  2. Pending 是结果，不是根因；节点资源不足只是其中一种可能。
  3. 对小范围变更使用服务端 dry-run，可以提前发现字段路径和 API 校验问题。
  4. 修复后的验证要覆盖控制面状态、Pod 就绪和业务可用性三个层次。
